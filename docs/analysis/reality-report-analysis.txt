  C1 — Health/Levy/Life Insurance Handlers Do Have Logic

  What the artefacts said: 0.0-highlights.md and 0.1-capability-inventory.md list health_insurance_flat, development_levy_flat, and life_insurance_rule as "Slot registered; no calculation logic." G3 in
  0.4-gap-outcomes.md repeats this.

  What the code actually does:

  # sequential_executor.py
  def _handle_health_insurance_flat(code, meta_json, results, salary_components, ctx):
      amount = Decimal(str(ctx.get("health_insurance_employee_amount", "0")))
      return {code: amount}

  def _handle_development_levy_flat(code, meta_json, results, salary_components, ctx):
      amount = Decimal(str(ctx.get("development_levy_amount", "0")))
      return {code: amount}

  def _handle_life_insurance_rule(code, meta_json, results, salary_components, ctx):
      rate = Decimal(str(ctx.get("life_insurance_employer_rate", "0")))
      return {code: (results.get("GROSS_PAY", Decimal("0")) * rate).quantize(...)}

  Context keys are populated in both payroll.py:177–179 and payroll_retry_service.py:171–173:

  health_insurance_employee_amount = Decimal(str(rules_jsonb.get("health_insurance", {}).get("employee_monthly_amount", "0")))
  development_levy_amount          = Decimal(str(rules_jsonb.get("development_levy", {}).get("monthly_amount", "0")))
  life_insurance_employer_rate     = Decimal(str(rules_jsonb.get("life_insurance", {}).get("employer_rate", "0")))

  Verdict: 🔵 Corrected — logic exists. The real problem is a rules_jsonb key name mismatch (see R1 below). The handlers are NOT orphaned.

  R1 — rules_jsonb Key Mismatch for Health Insurance and Development Levy

  Location: payroll.py:177–178, payroll_retry_service.py:171–172

  The code reads:
  - rules_jsonb.get("health_insurance", {}).get("employee_monthly_amount", "0")
  - rules_jsonb.get("development_levy", {}).get("monthly_amount", "0")

  The 0.5-business-rules-catalogue.md documents the actual seeded rules_jsonb structure as:
  { "health_insurance": { "employee_amount": 15000 }, "development_levy": { "amount": 1200 } }

  Keys "employee_amount" and "amount" do not match the keys the code reads ("employee_monthly_amount" and "monthly_amount"). No migration seeds these values with the correct key names into
  statutory_rule.rules_jsonb.

  Effect: Health insurance and development levy always compute ₦0 from the platform-level statutory rule. The only way they produce a non-zero value is if the workspace configures
  client_component_metadata overrides using the correct keys (employee_monthly_amount, monthly_amount). If no override exists, both deductions are silently zero and net pay is overstated.

  Classification: ⚠️  Partial — handler logic is correct; data pathway is broken at the rules_jsonb key name level.

  ---
  R2 — NHF Rate Key Mismatch in rules_jsonb

  Location: payroll.py:176, payroll_retry_service.py:170

  Code reads: rules_jsonb.get("nhf", {}).get("rate", "0.025")

  The catalogue shows stored key as: "nhf": { "employee_rate": "0.025" }

  The key "rate" does not exist in rules_jsonb; the stored key is "employee_rate". The code always falls through to the hardcoded default "0.025". This means:

  - The NHF rate in statutory_rule.rules_jsonb is never read — it cannot be changed via the DB without a code change.
  - The result is correct at the current statutory rate (2.5%) but the statutory_rule table does not actually control NHF.

  Classification: ⚠️  Partial — NHF calculates correctly by accident (hardcoded default matches rate), but statutory rule governance is broken.

  R3 — Life Insurance is an Employer Cost, Not a Net Pay Deduction

  Location: component_metadata migration c1d2e3f4a5b6, sequential_executor.py:423–435

  Life insurance component_class is "employer_cost", not "statutory_deduction". The _handle_net_formula handler only sums components where component_class == "statutory_deduction" when computing
  NET_PAY:

  total_deductions = sum(
      (v for k, v in results.items()
       if component_map.get(k, {}).get("component_class") == "statutory_deduction"),
      Decimal("0"),
  )

  Effect: Life insurance is computed (correctly) but does NOT reduce employee net pay — it is an employer-only cost per PRA 2014 s.9(3). The artefacts implied all three (health, levy, life) were payroll
   deductions; only health and levy are.

  Classification: ✅ Confirmed as correct — the design is intentional and legally accurate, but not clearly documented in the artefacts.

  R4 — Approve/Lock Transitions Hardcode performed_by = "admin@internal"

  Location: payroll_approval_service.py:95, 164

  approve_payroll_run() and lock_payroll_run() both write performed_by="admin@internal" to the audit log. Only mark_payroll_run_paid() accepts a real actor_id parameter.

  Effect: The audit trail for approval and locking cannot identify the individual operator — both always appear as the same internal system identity.

  Classification: ⚠️  Partial — audit log is written, but actor attribution is lost for two of three final-stage transitions.

  R5 — Execution Trace Written But Route Exposed Without Tracer Integration in Main Run

  Location: execution_trace_repo.py:11, payroll.py:866

  The GET /{wid}/payroll/runs/{run_id}/timeline route queries execution_trace table correctly. However, the main run_payroll path in payroll.py does not instantiate or pass a tracer — the
  executor.py:144 shows tracer = tracer or NULL_TRACER and NULL_TRACER writes nothing. The execution_tracer.py is available as a decorator/context manager, but is not wired into the primary API-driven
  run path.

  Effect: The timeline endpoint exists and functions, but returns empty results for all runs triggered through the normal API because the tracer is not active. execution_trace remains empty in
  production.

  Classification: ⚠️  Partial — read path exists; write path is not connected.

  R6 — statutory_rule.state (DRAFT/LIVE) Never Checked at Rule Resolution

  Location: payroll.py:142–150 — statutory rule query

  The query resolves rules by effective_from <= period_end with no WHERE state = 'LIVE' filter. A DRAFT statutory rule with a recent effective_from will be selected and applied to a live run.

  Classification: ⚠️  Partial — the state column exists and is stored; it is never used as a gate.
