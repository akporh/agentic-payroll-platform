## Category 1: Multi-Client Isolation & Data Security

### Rule 1.1: Workspace-Based Tenant Isolation
Every client is assigned a dedicated workspace. No employee, salary definition, payroll rule, or payroll result can cross workspace boundaries.

**Why This Matters:** A data leak between clients is a catastrophic compliance failure for a bureau.

**Real-World Example:** Dangote Cement and ABC Bank are both bureau clients. An HR manager at Dangote cannot retrieve, view, or trigger payroll for ABC Bank employees.

**Implementation:** Every core table carries `workspace_id` FK. Every query filters by `workspace_id` as its first condition.

**Status: Fully Implemented**

---

### Rule 1.2: Workspace Must Be Activated Before Payroll Can Run
A workspace must be fully configured and set to LIVE before any payroll run can be created.

**Why This Matters:** Prevents payroll execution against incomplete client setup — missing salary definitions, unloaded employees, or absent statutory configuration.

**Implementation:** DB trigger `enforce_workspace_live_before_payroll` fires on BEFORE INSERT on `payroll_run`. If `workspace.status ≠ 'LIVE'`, the run is rejected at DB level.

**Status: Fully Implemented**

---

### Rule 1.3: Employee Numbers Are Unique Per Client
No two employees within the same client workspace can share an employee number.

**Implementation:** Unique DB index `uq_employee_number_per_workspace ON employee(workspace_id, employee_number)`.

**Status: Fully Implemented**

---

### Rule 1.4: One Active Employment Contract Per Employee
An employee may have only one active (open-ended) contract at any point in time.

**Why This Matters:** Two active contracts produce double calculation — double salary, double pension, double PAYE.

**Implementation:** Partial unique index `uq_employee_active_contract ON employee_contract(employee_id) WHERE end_date IS NULL`.

**Status: Fully Implemented**

---

### Rule 1.5: Statutory Rules Are Platform-Level, Not Client-Specific
Tax bands, pension rates, and NHF rates are managed by the bureau at platform level. Clients cannot modify them.

**Implementation:** `statutory_rule` table has no `workspace_id` — shared across all clients. Lookup by `country_code`.

**Status: Fully Implemented**
**Gap:** No API or UI for the bureau to manage statutory rules. Updates require direct database intervention.

## Category 2: Workforce & Compensation Management

### Rule 2.1: Salary Structures Are Defined by Grade, Not by Individual
Compensation is defined as named salary templates assigned to employees via contracts.

**Implementation:** `salary_definition.components_jsonb` holds the template. `employee_contract.salary_definition_id` links employee to template.

**Status: Fully Implemented**
**Gap:** Salary definition effective dates (`effective_from`, `effective_to`) exist in schema but are never checked at run time.

---

### Rule 2.2: Salary Definitions Cannot Be Modified After a Paid Run
Once a run reaches PAID, salary structures used in that run are permanently frozen.

**Implementation:** DB trigger `prevent_salary_definition_change_if_used` on BEFORE UPDATE/DELETE of `salary_definition`. Queries whether any linked `payroll_run.status = 'PAID'`.

**Status: Fully Implemented**

---

### Rule 2.3: Salary Definitions Must Contain Mandatory Components
Every salary definition must include BASIC, HOUSING, and TRANSPORT — the three legally recognised pensionable elements under PRA 2014.

**Implementation:** `hard_validator.py` checks during onboarding validation.

**Status: Partially Implemented**
**Gap:** Validation only runs during onboarding. Direct DB insert bypasses it. No CHECK constraint at DB level.

---

### Rule 2.4: Payroll Rules Are Versioned and Effective-Dated
Earning/deduction rules are published in versioned rule sets. Historical inputs are calculated at the rate in effect when they were incurred.

**Implementation:** `rule_set` and `rule_set_item` tables with `effective_from`. Cross-period rate resolution in `rule_evaluator.py`.

**Status: Partially Implemented**
**Gap:** Silent fallback to current rate if no historical rule set covers the input's reference month. Fallback not flagged in rule_trace.

## Category 3: Payroll Calculation & Financial Accuracy

### Rule 3.1: All Monetary Calculations Use Decimal Arithmetic
No floating-point arithmetic anywhere. All amounts computed as exact decimals, rounded to 2dp using ROUND_HALF_UP.

**Implementation:** Python `Decimal` type throughout all rule handlers. Every monetary output calls `.quantize(Decimal("0.01"), rounding=ROUND_HALF_UP)`.

**Status: Fully Implemented**

---

### Rule 3.2: Payroll Components Execute in a Defined, Immutable Order
BASIC → HOUSING → TRANSPORT → GROSS_PAY → PENSION_EMPLOYEE → TAXABLE_INCOME → PAYE → NHF → NET_PAY. Always in this exact sequence.

**Why This Matters:** PAYE applied to gross (before pension relief) produces higher tax than PAYE on taxable income. The correct sequence is mandated by FIRS and PRA 2014.

**Implementation:** `sequential_executor.py` sorts components by `execution_priority` ASC before dispatch. Order cannot be changed at runtime.

**Status: Fully Implemented**

---

### Rule 3.3: PAYE Is Calculated on Taxable Income, Not Gross Pay
PAYE is applied to gross pay reduced by pension employee contribution and any qualifying rent relief.

**Implementation:** `TAXABLE_INCOME` component (priority 300) computes `GROSS_PAY − PENSION_EMPLOYEE − RENT_RELIEF`. PAYE handler (priority 400) reads `TAXABLE_INCOME`.

**Status: Fully Implemented**

---

### Rule 3.4: Pension Is Calculated on the Statutory Pensionable Base (PRA 2014)
Employee pension (8%) and employer pension (10%) computed on BASIC + HOUSING + TRANSPORT.

**Implementation:** `_handle_pension_rule` in `sequential_executor.py`. Default base is BASIC + HOUSING + TRANSPORT. Client-configurable via `client_component_metadata`.

**Status: Fully Implemented**

---

### Rule 3.5: NHF Is Calculated at 2.5% of Basic Salary
National Housing Fund deducted at 2.5% of BASIC per the NHF Act.

**Implementation:** `_handle_nhf_rule`; base component configurable via `component_metadata.metadata_json.base_component`, defaulting to BASIC.

**Status: Fully Implemented**

---

### Rule 3.6: Proration Applies to Mid-Period Hires and Terminations
An employee who joins or leaves mid-period receives proportional salary for days worked.

**Implementation:** `compute_hire_termination_factor()` in `period_context.py`. Returns Decimal [0,1] based on `contract_start`/`contract_end` vs period.

**Status: Fully Implemented**

---

### Rule 3.7: Three Statutory Components Lack Calculation Logic
Health Insurance Employee, Development Levy, and Life Insurance handlers are registered but produce ₦0.

**Status: Not Implemented**
**Action Required:** Bureau must confirm scope and commission implementation before go-live.

## Category 4: Period Management & Temporal Correctness

### Rule 4.1: One Payroll Run Per Client Per Period
Only one regular payroll run can exist per client per period.

**Implementation:** Unique DB index `uq_payroll_run_period ON payroll_run(workspace_id, period_start, period_end)`. Rejection returned as HTTP 409.

**Status: Fully Implemented**

---

### Rule 4.2: Variable Inputs Can Be Staged Against a Specific Past Month
Overtime, absence, or bonus records submitted with a `reference_date` are held and picked up by the next run covering that period.

**Implementation:** `payroll_input.reference_date`; claiming logic `WHERE reference_date IS NULL OR reference_date <= period_end`. Rate resolution uses rule set effective on that month.

**Status: Partially Implemented**
**Gap:** Silent fallback to current rate if no historical rule set covers the reference month.

---

### Rule 4.3: Correct Statutory Rules Applied Based on Pay Period, Not Today
Tax bands and rates selected based on `period_end`, not execution date.

**Implementation:** `SELECT statutory_rule WHERE effective_from <= period_end ORDER BY effective_from DESC, version DESC LIMIT 1`. Resolved `statutory_effective_date` stored on `payroll_run` for retry determinism.

**Status: Partially Implemented**
**Gap:** Non-deterministic if two statutory rule rows share same `effective_from`.

---

## Category 5: Run Lifecycle & Authorisation Controls

### Rule 5.1: Mandatory Authorisation Chain
DRAFT → CALCULATING → CALCULATED → APPROVED → LOCKED → PAID. No status can be skipped or reversed.

**Implementation:** `state_machine.py` validates before DB write. DB trigger `validate_payroll_status_transition` enforces rank-based progression. `SELECT FOR UPDATE` prevents concurrent state changes.

**Status: Fully Implemented**
**Gap:** Approve, Lock, Mark Paid have no UI buttons — API-only. Bureau operators cannot complete the authorisation chain from the web interface.

---

### Rule 5.2: A Paid Payroll Run Is Permanently Immutable
No data associated with a PAID run can be changed by any user, application, or database process.

**Implementation:** DB trigger `prevent_paid_payroll_run_update` (BEFORE UPDATE/DELETE when status='PAID'). DB trigger `prevent_payroll_result_mutation` (when parent run is CALCULATED/APPROVED/LOCKED/PAID).

**Status: Fully Implemented**

---

### Rule 5.3: Results Are Frozen at the Calculation Stage
Individual employee results cannot be modified once run moves past PARTIAL/CALCULATING.

**Implementation:** `prevent_payroll_result_mutation` blocks when parent run status ∈ {CALCULATED, APPROVED, LOCKED, PAID}.

**Status: Fully Implemented**

---

### Rule 5.4: Calculation Snapshots Are Immutable
The full record of rules, rates, and configuration used to produce each payslip is frozen at calculation and can never be changed.

**Implementation:** DB trigger `prevent_snapshot_update` (BEFORE UPDATE OF `calculations_snapshot_json`). Separate trigger for `payroll_run.rules_context_snapshot`.

**Status: Fully Implemented**

---

## Category 6: Retry & Recovery

### Rule 6.1: Failed Calculations Can Be Retried Without Affecting Successful Ones
Only FAILED employees are reprocessed. Successful results are untouched.

**Implementation:** `payroll_retry_service.py` PER_EMPLOYEE strategy: DELETE old FAILED row (guarded by `status='FAILED'`), recalculate, INSERT new row.

**Status: Fully Implemented**
**Gap:** Run-level totals not recalculated after retry.

---

### Rule 6.2: Retry Uses Original Run's Rules, Not Current Rates
Same statutory rates, tax bands, and payroll rules as the original run — not live values.

**Implementation:** `payroll_retry_service.py:130–152` — reads `statutory_effective_date` from original run; uses frozen `rules_context_snapshot` for historical rule sets.

**Status: Fully Implemented**

---

### Rule 6.3: Retry Blocked on Authorised or Paid Runs
Only PARTIAL runs eligible for retry.

**Implementation:** `payroll_retry_service.py:556–596` — explicit guards reject PAID, APPROVED, LOCKED.

**Status: Fully Implemented**

---

## Category 7: Reconciliation & Disbursement

### Rule 7.1: Bureau Must Confirm Actual Disbursement Against Calculated Total
Actual amount disbursed is recorded and compared against calculated total net pay. Result: MATCHED or MISMATCH.

**Implementation:** `POST /payroll/reconcile` → `payroll_reconciliation` INSERT.

**Status: Partially Implemented**
**Gaps:**
- Reconciliation accepted at any run state (should require LOCKED/PAID)
- MISMATCH triggers no follow-up action
- Duplicate submission causes unhandled 500 error
- expected_total is stale after retry

---

### Rule 7.2: Disbursement Files Cannot Be Generated From the System
The system has no mechanism to produce bank payment files, PAYE remittance schedules, or pension schedules from the UI or API.

**Implementation:** Export functions exist (`net_pay_export.py`, `paye_export.py`, `payroll_register_export.py`) but have no API endpoints or UI.

**Status: Not Wired**
**Action Required:** Export endpoints must be created and integrated into post-LOCKED/PAID workflow.

---

## Category 8: Audit, Traceability & Compliance

### Rule 8.1: Every Status Transition Is Recorded
audit_log entry written on every state change: previous state, new state, actor, timestamp.

**Status: Partially Implemented**
**Gaps:** `performed_by` hardcoded to `"admin@internal"`. Retry operations not logged. No read endpoint for audit trail.

---

### Rule 8.2: Every Payslip Contains a Full Calculation Breakdown
`calculations_snapshot_json` and `component_trace_jsonb` stored per employee result.

**Status: Partially Implemented**
**Gaps:** `component_trace_jsonb` absent for legacy executor path. No UI renders these fields. No employee payslip view.

---

### Rule 8.3: Inputs Carry Source Attribution
`payroll_input.source` records origin (manual, bulk upload, API).

**Status: Partially Implemented**
**Gap:** `source` written at creation; never read in any downstream process or UI display.
