 Gap Outcomes                    
                                                                                                                                                                                                                                                                             
  ---                                                   
  G1 — Retry Does Not Recalculate Run Totals                                                                                                                                                                                                                                 
                                                        
  What the logic does: After retry, payroll_run.total_gross_pay, total_deduction, total_net_pay are left at their pre-retry values. Only payroll_result rows are updated.
                                                                                                                                                                                                                                                                             
  Blocked outcome: Any consumer of run-level aggregates — including the reconciliation flow and the GET /payroll/runs/{workspace_id} list endpoint — reads stale figures after a retry.                                                                                      
                                                                                                                                                                                                                                                                             
  Classification: Admin/internal — Incomplete                                                                                                                                                                                                                                
                                                        
  ▎ The totals shown to an operator after a successful retry are wrong. Reconciliation built on those totals will compare the wrong expected figure against the actual disbursed amount.                                                                                     
                                                        
  ---                                                                                                                                                                                                                                                                        
  G2 — Retry Not Audited                                
                        
  What the logic does: Retry rewrites payroll_result rows and updates payroll_run.status but writes no audit_log or event_store entry for the retry operation itself.
                                                                                                                                                                                                                                                                             
  Blocked outcome: No traceable record that a retry occurred, who triggered it, when, or what changed. The audit trail shows only the original run transitions (DRAFT→CALCULATING→CALCULATED/PARTIAL).                                                                       
                                                                                                                                                                                                                                                                             
  Classification: Backend-only — Incomplete                                                                                                                                                                                                                                  
                                                        
  ▎ From the audit log alone, a retried PARTIAL run that resolves to CALCULATED is indistinguishable from a clean first-run CALCULATED. The retry event is a system capability with no persistent evidence.                                                                  
   
  ---                                                                                                                                                                                                                                                                        
  G3 — Health Insurance / Development Levy / Life Insurance Handlers Unimplemented
                                                                                  
  What the logic does: Component metadata rows and handler slots exist at priorities 420, 430, 440 in sequential_executor.py. No calculation logic is registered for health_insurance_flat, development_levy_flat, or life_insurance_rule.
                                                                                                                                                                                                                                                                             
  Blocked outcome: These three statutory deductions produce 0 (or are absent from results) for every employee on every run, silently. net_pay is overstated by the sum of these missing deductions.                                                                          
                                                                                                                                                                                                                                                                             
  Classification: Admin/internal — Incomplete                                                                                                                                                                                                                                
                                                        
  ▎ The component slots exist in metadata and appear in component priority order, but their contributions to deductions_jsonb and net_pay are zero. No error is raised; no warning is surfaced. An operator reviewing results has no signal that these lines are missing.    
   
  ---                                                                                                                                                                                                                                                                        
  G4 — Reconciliation Accepts Any Run State             
                                                                                                                                                                                                                                                                             
  What the logic does: reconcile_payroll_run() runs a SELECT payroll_run.total_net_pay and immediately inserts a reconciliation record with no check on payroll_run.status.
                                                                                                                                                                                                                                                                             
  Blocked outcome: A reconciliation record can be created against a DRAFT, CALCULATING, or PARTIAL run — states where the totals are either zero, provisional, or stale. The reconciliation outcome (MATCHED/MISMATCH) is meaningless at those states.                       
                                                                                                                                                                                                                                                                             
  Classification: Admin/internal — Incomplete                                                                                                                                                                                                                                
                                                        
  ▎ Reconciliation is a post-payment check. Without a state gate (e.g. status must be LOCKED or PAID), the result is unreliable and the MATCHED status may be a false positive.                                                                                              
   
  ---                                                                                                                                                                                                                                                                        
  G5 — Duplicate Reconciliation Raises Unhandled 500    
                                                    
  What the logic does: A second POST /payroll/reconcile for the same payroll_run_id causes a DB IntegrityError (unique constraint on payroll_reconciliation.payroll_run_id). This is not caught; it propagates as an HTTP 500.
                                                                                                                                                                                                                                                                             
  Blocked outcome: An operator who reconciles twice (e.g. correction run) gets an opaque server error instead of a 409 Conflict or an update path.                                                                                                                           
                                                                                                                                                                                                                                                                             
  Classification: Admin/internal — Incomplete                                                                                                                                                                                                                                
                                                        
  ▎ The error is externally visible as a 500 but the message exposes DB internals. There is also no update path — a MISMATCH cannot be corrected to MATCHED without direct DB manipulation.                                                                                  
   
  ---                                                                                                                                                                                                                                                                        
  G6 — client_component_metadata Has No Write API       
                                                 
  What the logic does: The client_component_metadata table is read during payroll execution to apply workspace-level overrides (suppress components, set flat amounts). Rows can only be written via POST /onboarding/commit.
                                                                                                                                                                                                                                                                             
  Blocked outcome: A workspace cannot update its component overrides after initial onboarding without re-running the full onboarding commit. There is no endpoint to add, update, or remove individual overrides post-setup.                                                 
                                                                                                                                                                                                                                                                             
  Classification: Backend-only — Incomplete                                                                                                                                                                                                                                  
                                                        
  ▎ The override mechanism is fully wired into the execution path but is effectively one-time-write from the operator's perspective. Post-onboarding override changes require direct DB access.                                                                              
   
  ---                                                                                                                                                                                                                                                                        
  G7 — Statutory Rule Temporal Tie-Breaking Non-Deterministic
                                                             
  What the logic does: Rule selection uses ORDER BY effective_from DESC, version DESC LIMIT 1. If two statutory rule rows share the same effective_from date, version is the tiebreaker. But version is an application-managed integer with no uniqueness constraint across
  rows.                                                                                                                                                                                                                                                                      
   
  Blocked outcome: Two published rules with the same effective_from but different version values will resolve consistently (higher version wins). However, if version values are also equal or unmanaged, the result is undefined and depends on insertion order.            
                                                        
  Classification: Backend-only — Incomplete                                                                                                                                                                                                                                  
                                                        
  ▎ In practice this requires a data entry error to trigger, but when it does, the wrong statutory rates are applied to the entire run silently.                                                                                                                             
   
  ---                                                                                                                                                                                                                                                                        
  G8 — Workspace country_code Not Validated at Creation 
                                                       
  What the logic does: POST /workspace accepts any string for country_code with no check against the statutory_rule table.
                                                                                                                                                                                                                                                                             
  Blocked outcome: A workspace created with country_code='ZZ' (or a typo) will fail silently at payroll run time — the statutory rule query returns no rows, and the run either errors or proceeds with empty rates depending on how the null is handled.                    
                                                                                                                                                                                                                                                                             
  Classification: Admin/internal — Incomplete                                                                                                                                                                                                                                
                                                        
  ▎ The failure is deferred to run time. An operator will only discover the misconfiguration when attempting the first payroll run, not at workspace creation.                                                                                                               
                                                        
  ---                                                                                                                                                                                                                                                                        
  G9 — daily_rate_deduction Has No Bounds Check         
                                               
  What the logic does: Deduction amount = (gross / working_days) × absent_days. There is no check that absent_days ≤ working_days.
                                                                                                                                                                                                                                                                             
  Blocked outcome: An input of absent_days = 30 in a 22-working-day month produces a deduction larger than gross pay, resulting in a negative net_pay. This is persisted to payroll_result without error.                                                                    
                                                                                                                                                                                                                                                                             
  Classification: Backend-only — Incomplete                                                                                                                                                                                                                                  
                                                        
  ▎ The calculation is internally consistent but produces an economically invalid result. Negative net pay propagates into totals, reconciliation, and any export outputs without any flag.                                                                                  
   
  ---                                                                                                                                                                                                                                                                        
  G10 — Cross-Period Input Rate Resolution Silent Fallback
                                                          
  What the logic does: When a payroll_input row carries a reference_date from a prior period, rule_evaluator.py looks up the historical rule set effective on that date. If no matching rule set is found, it silently falls back to the current period's rate.
                                                                                                                                                                                                                                                                             
  Blocked outcome: Overtime or allowance inputs from a month where the rate was different are calculated at the wrong (current) rate. No warning, no flag in rule_trace.                                                                                                     
                                                                                                                                                                                                                                                                             
  Classification: Backend-only — Incomplete                                                                                                                                                                                                                                  
                                                        
  ▎ The rule_trace entry does include a resolution_source field, but the fallback path does not mark the resolution as a fallback — it appears identical to a valid historical resolution.                                                                                   
   
  ---                                                                                                                                                                                                                                                                        
  G11 — Bulk Payroll Input Has No Deduplication         
                                               
  What the logic does: POST /{workspace_id}/payroll/inputs (bulk) inserts each row independently. There is no uniqueness constraint on (workspace_id, employee_id, input_code, reference_date) in payroll_input.
                                                                                                                                                                                                                                                                             
  Blocked outcome: Re-submitting the same bulk CSV twice doubles every input. Both rows are unclaimed and both will be claimed by the next payroll run, doubling the quantity used in calculation.                                                                           
                                                                                                                                                                                                                                                                             
  Classification: Admin/internal — Incomplete                                                                                                                                                                                                                                
                                                        
  ▎ No error is raised on the second submission. The operator receives a 200 with a count of inserted rows. The doubled inputs are invisible until the payroll run executes.                                                                                                 
   
  ---                                                                                                                                                                                                                                                                        
  G12 — Legacy Executor Path Produces No component_trace_jsonb
                                                                                                                                                                                                                                                                             
  What the logic does: When executor.py falls back to the legacy path (no component_metadata provided), it calls calculate_gross() + calculate_net_pay() directly. component_trace_jsonb is NULL in the resulting payroll_result row.
                                                                                                                                                                                                                                                                             
  Blocked outcome: For any employee processed via the legacy path, there is no per-component breakdown in the result. Only gross, paye, and net_pay are available.                                                                                                           
                                                                                                                                                                                                                                                                             
  Classification: Backend-only                                                                                                                                                                                                                                               
                                                        
  ▎ The legacy path is used during retry in certain edge cases and by any caller that omits component_metadata. The missing trace means those results cannot be audited at the component level, and the calculations_snapshot_json may also be sparse.
