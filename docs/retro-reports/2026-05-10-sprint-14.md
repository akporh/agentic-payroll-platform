## Issues Caught Late

### Issue 1 — simulate_payroll.py "propagates automatically" claim was false

**What happened:**
The plan stated: *"payroll_retry_service.py and simulate_payroll.py both call execute_single_employee_payroll → _run_sequential, so the executor fix propagates automatically. No additional changes needed."*

This was wrong. `simulate_payroll.py` calls `run_sequential_payroll` from `sequential_executor.py` directly — it does not go through `executor.py::_run_sequential` at all. The per-component proration loop added to `executor.py` was therefore never reached when running the simulate script. The integration test (running the script for employee df283984) showed unprorated values — the gap was caught there.

**Stage caught:** Tester — integration test  
**Should have been caught:** Architect — plan mode, before writing "no changes needed"

**Why it slipped:**
The plan made an assumption about the call chain without verifying it. A memory entry (`project_simulation_scripts_no_proration.md`) explicitly documented the gap. The plan contradicted that memory entry without addressing it.

**Fix applied:**
Added `ec.end_date` to `_load_contract_details`, inserted the per-component proration loop (mirroring `executor.py`) after `apply_payroll_rules` in `simulate_payroll.py`, imported `compute_hire_termination_factor`, and added a `HIRE / TERMINATION PRORATION` display section.

**Skill update — `architect/SKILL.md`:**
New section: *"Propagates Automatically" Claims Must Be Verified* — grep the script's call chain to confirm it calls through the changed function, not a parallel entry point. If memory contains a known gap about that script, the plan must explicitly address it.

**Skill update — `tester/SKILL.md`:**
New check: *"Propagates automatically" integration verification* — for each script named in the plan as "no changes needed," run it for a case that exercises the new behaviour and assert the new behaviour is present in the output, not just that the script ran without crashing.

### Issue 2 — PRORATION_OPTIONS values didn't match DB domain strings

**What happened:**
`PRORATION_OPTIONS` in `WorkspaceConfig.tsx` had:
```tsx
{ value: 'FULL_MONTH',     label: 'Full Month' }
{ value: 'CALENDAR_DAYS',  label: 'Calendar Days' }
{ value: 'WORKING_DAYS',   label: 'Working Days' }
```
The DB stores `'fixed_30'` / `'calendar_days'` / `'work_days'`. Any user interaction with the proration dropdown would have written an invalid string to the DB (e.g. `'FULL_MONTH'`). The PATCH endpoint writes whatever string it receives; the engine's `else`-branch silently falls back to `work_days` for unrecognised values. No error was surfaced.

The smoke test confirmed this was destructive: PATCHING with `proration_strategy: 'FULL_MONTH'` successfully overwrote the existing `'fixed_30'` value in the DB.

**Stage caught:** Tester — API smoke test  
**Should have been caught:** Frontend-designer or Tester, during the P2 implementation review

**Why it slipped:**
P2 was treated as a small cosmetic fix. The option values weren't cross-referenced against DB domain values or the backend enum.

**Fix applied:**
`PRORATION_OPTIONS` corrected to `work_days` / `calendar_days` / `fixed_30`. Both useState defaults and useEffect fallbacks updated to match.

**Skill update — `tester/SKILL.md`:**
New check: *Select option values must match DB domain exactly* — for every `<select>` backed by a stored enum/strategy column, verify each `value=` attribute against the actual DB-stored strings, not display labels or legacy values.

## What Went Well

**1. Unit tests designed directly against acceptance criteria**
HP-1/HP-2/HP-3 from the story file became named test cases (`test_march_4_hire_hp1`, `test_march_4_hire_hp2`, `test_hp3_explicit_matches_default`). Each would catch a regression in its strategy branch immediately. The cross-strategy comparison test (`test_three_strategies_give_distinct_factors_for_march_4_hire`) verifies all three branches produce distinct, expected values from a single test.

**2. Memory loop is working — gap was pre-documented**
`project_simulation_scripts_no_proration.md` existed before this sprint and accurately described the problem. The retro learning system from prior sprints surfaced the right fact at the right moment (during the tester's integration test review). The failure was in plan mode not consulting it — not in the memory system itself.

**3. API-level smoke test as browser substitute**
With no display available for UI testing, the smoke test confirmed the P2 bug via a PATCH round-trip:
- Baseline: `proration_strategy = 'fixed_30'`
- Fixed form (correct): PATCH with `'fixed_30'` → DB still `'fixed_30'` ✅
- Old bug (demonstrated): PATCH with `'FULL_MONTH'` → DB `'FULL_MONTH'` (destructive confirmed)
- Restored to correct state

This three-step pattern (baseline → action → verify; then inverse → demonstrate breakage) is worth repeating for any form-save test where silent overwrite is the failure mode.
