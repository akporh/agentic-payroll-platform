## Test Suite Fixes (Sprint 18 side-effect — UNIQUE constraint on statutory_rule)

MIG-18 session applied `uq_statutory_rule_country_effective` UNIQUE constraint to the DB (pre-existing migration). The UAT cleanup during this session consolidated the live NG rule at `2000-01-01`, causing 6 test fixture inserts at that date to collide. All 6 were fixed:

| File | Old effective_from | New effective_from | Expected values corrected? |
|---|---|---|---|
| test_payroll_lock_and_approval.py | `1999-01-01` → `2025-06-01` → | `2026-04-01` | Yes — EXPECTED_NET 578,273.33 → 590,773.33 (no NHF workspace rule) |
| test_payroll_pipeline_e2e.py | `2000-01-01` | `2026-02-01` | Yes — EXPECTED_NET/NHF updated |
| test_payroll_partial_run_e2e.py | `2000-01-01` | `2026-03-01` | Yes — EXPECTED_NET updated |
| test_payroll_retry.py | `2000-01-01` | `2026-02-15` | Yes — EXPECTED_NET_A/B updated |
| test_payroll_run_snapshot_immutable.py | `2000-01-01` | `2026-03-15` | No assertions on net_pay |
| test_payroll_snapshot_integrity.py | `2000-01-01` | `2026-04-15` | Checks identity (before==after) |

Root cause: tests assumed `ORDER BY version DESC` picked the winner; actual query uses `ORDER BY effective_from DESC`. Dates prior to seed (`2026-01-01`) always lose. New dates all land in `(2026-01-01, 2026-05-31]`.

Additional test fix: `test_payroll_run_readiness.py` — `end_date='2020-01-01'` against a contract with `start_date=CURRENT_DATE` (2026-05-31) triggered MIG-18-B's no-overlap range check ("range lower bound must be less than or equal to range upper bound"). Fixed by adding `contract_start='2025-01-01'` to onboarding payload and changing expiry to `'2025-12-31'`.

Additional route fix: `backend/api/routes/payroll.py` — runs created without explicit `period_start`/`period_end` saved NULL to DB, blocking the P1-3 retry guard. Added fallback to `period_ctx.period_start`/`period_ctx.period_end` after `build_period_context` so runs always have period dates persisted.
