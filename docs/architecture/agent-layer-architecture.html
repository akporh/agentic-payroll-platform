As-Is Gap Register
  • GAP-1
    No authentication — performed_by = "admin@internal" hardcoded in service layer; workspace_id from request body
    Affects: every route, every audit log row, every event_store entry
    Critical
  • GAP-2
    Event store is write-only — no consumer, no dispatcher, no reaction logic
    events exist in the DB but have zero downstream effect
    Critical
  • GAP-3
    Missing events: reconciliation MISMATCH, employee enrollment, status change (ACTIVE/INACTIVE), payroll input submitted
    Proactive agents in Phase 2B cannot trigger without these events
    Critical
  • GAP-4
    Events written post-commit (fire-and-forget) — process crash between commit and event write causes silent event loss

  • GAP-5
    No background task infrastructure — payroll execution is a blocking HTTP call; no async processing
    APScheduler required for event consumer worker
    High
  • GAP-6

    Required before Phase 2B proactive agents

X Track X — Proactive Agents (Phase 2B)

Event-triggered agents that surface issues before operators discover them manually. First write-capable phase — all mutations require structured confirmation.

Pre-condition before Phase 2B sprint planning: Confirmation protocol must be fully specified. Mechanism: structured confirmation UI component (not chat reply) showing exact record ID, field, and new value — backed by service-layer pending_action_id. Natural-language "yes" in chat is not sufficient for financial mutations.

Technology Decisions
DecisionChoiceRationale
Primary LLM (chat) claude-sonnet-4-6 Speed + cost appropriate for conversational latency
Primary LLM (investigation agents) claude-opus-4-8 Deeper reasoning for component trace + reconciliation investigation
Fallback LLM GPT-4o Auto-reroute on Anthropic unavailability; silent to operator; logged in session
LLM routing Vercel AI Gateway Unified API; zero-data-retention mode; per-workspace observability; model fallbacks
Task queue APScheduler (no Celery) Current scale does not justify Celery operational overhead; single-worker constraint documented; distributed lock required for multi-process
Conversation history Ephemeral session + agent_session_log No full replay; audit trail only; 7-year retention for payroll dispute resolution
workspace_id source JWT only Never from message content — prompt injection could shift workspace context with catastrophic cross-client data exposure in a multi-tenant bureau
Tool granularity Narrow domain-shaped tools Not generic SQL-like queries — keeps capability surface auditable; prevents LLM constructing arbitrary data access patterns
Ordering + Gate Rules
RuleDetail
Track P before all of Phase 2 No Phase 2A sprint begins until JWT auth is complete and verified
V1 + V2 together Transactional outbox and event gap closure ship in the same sprint — outbox alone without new events does not unblock proactive agents
Rate limiting (W3) ships in Sprint A2 Not deferred to Sprint A4 — unbounded Claude API calls before rate limiting creates cost and DoS risk
agent_session_log (W5) ships after Track P Placeholder operator_id audit trail is worse than none
Phase 2B confirmation protocol Must be fully specified before Phase 2B sprint planning — mechanism is a structured UI component with service-layer pending_action_id, not a chat reply
Security Invariants (non-negotiable)
  • ⊗
    workspace_id from JWT only
    Never from message content, request body, or tool call parameters
  • ⊗
    PII stripped before LLM context
    name, TIN, RSA pin, bank account, employee_number — removed at tool serialisation layer; UUID-only reference in LLM context
  • ⊗
    Tool results as structured JSON envelopes
    Freeform string fields (descriptions, notes) wrapped with labels — never raw interpolation into system prompt
  • ⊗
    No write tools in Phase 2A
    Read-only throughout Tracks P and W. Write capability introduced only in Phase 2B with pending_action_id + structured confirmation
  • ⊗
    Decimal serialisation in LLM context = string
    NOT float (which is lossy and produces "₦124,999.9999999998"). The existing project JSONB rule (float) does NOT apply to LLM context
