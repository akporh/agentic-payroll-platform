01 — Overview

Payroll Execution Pipeline

The payroll execution pipeline transforms an employee's salary definition, workspace-configured rules, and statutory requirements into a final payslip — calculating gross pay, overtime, deductions, PAYE, pension, NHF, and net pay in a deterministic, auditable sequence. Every run is frozen at creation and can be reproduced exactly from the stored snapshot.

ℹ️
Three layers work together. The application layer (payroll_run_service.py) orchestrates. The domain layer (rule_evaluator.py + sequential_executor.py) calculates. The infrastructure layer persists results. Domain code never touches the database — all DB access is in the infra layer.

The Three Components

ComponentLayerResponsibilitySource file
payroll_run_service Application Orchestrates the full run: validates, calls domain, persists, traces backend/application/payroll_run_service.py
rule_evaluator Domain Applies workspace rules (OT, absences, bonuses) to base salary before statutory deductions backend/domain/payroll/rule_evaluator.py
sequential_executor Domain Applies statutory deductions (PAYE, pension, NHF, rent relief) in strict priority order backend/domain/payroll/sequential_executor.py

⚠️
Proration fires BEFORE rule evaluation. This means OT and absence rules in rule_evaluator receive already-prorated salary_components. For mid-period hires: the daily rate used in absence deductions is computed from the prorated base, not the full monthly salary. This is the correct sequencing (fixed in Sprint 14) but it means daily rates for partial-month employees are lower than expected if you compare to a non-prorated reference.

State semantics and guards

StateWhat it meansAllowed nextGuardExports available
DRAFT Run created; context snapshot frozen; no execution started CALCULATING — None
CALCULATING Engine processing employees; may fail for some CALCULATED, PARTIAL — None
PARTIAL ≥1 employee FAILED; retryable CALCULATED (after retry) Only FAILED employees reprocessed; SUCCESS results never touched None
CALCULATED All employees passed; awaiting human approval APPROVED — Payroll detail export
APPROVED Authorised; no retry or recalculation possible LOCKED Retry blocked (Python guard). Contract changes blocked (D-ARCH-1) Payroll detail export
LOCKED Immutable at Python layer; ready for disbursement PAID Python guard on all mutation routes Bank upload, PAYE remittance, pension schedule
PAID Terminal. Disbursed. None DB trigger trg_prevent_paid_run_update raises exception on any UPDATE All exports
🔒
Security note: APPROVED → immutable is enforced at the Python layer (not DB). A direct DB UPDATE bypasses this guard. Only the DB trigger on PAID provides true DB-level immutability. For APPROVED, the Python guard is the only enforcement. Recommend adding a DB CHECK or trigger for APPROVED state in a future hardening sprint.

07 — Edge Cases

Edge Cases

Hidden scenarios that produce unexpected behaviour. Each case is verified against the actual source code.

1
INACTIVE employee with live enrolled contract
Operator High
▼
Setup
# Employee manually set to INACTIVE employee.status = 'INACTIVE' employee_contract.end_date = NULL # still open employee_contract.salary_definition_id = '<uuid>' # enrolled
Current Behaviour

Employee is excluded from payroll runs — the engine checks WHERE status = 'ACTIVE'. The contract row exists but is unused.

On the Employees page, this employee appears in the "Deactivated" sub-section (not in payroll groups). An AlertBanner warns the operator if they have a live contract.

Risk & Mitigation

Risk: An operator on suspension/maternity leave may be INACTIVE intentionally. If the operator re-activates the employee mid-period without adjusting contract dates, the employee enters payroll immediately with a full-month salary (no proration for the INACTIVE period). Payroll for the weeks they were inactive is unintentionally included.

Mitigation: The AlertBanner on the Employees page surfaces this state. The operator must manually set the correct contract_start or end the current contract and create a new one when re-activating. Garden leave and suspension scenarios are not yet modelled — end_date must be set to the last date salary is paid.

Enforced at: backend/api/routes/payroll.py:735 (status = 'ACTIVE' guard)
2
Ended contract with orphaned payroll_input rows
Operator High
▼
Setup
contract.end_date = '2026-02-28' payroll_input rows exist for March 2026 period_start = '2026-03-01'
Current Behaviour

Employee excluded from the run (contract end < period_start). The payroll_input rows remain in the database, unclaimed.

They are surfaced in the "deactivated_with_inputs" count visible on the PayrollResults page banner.

Risk & Mitigation

Risk: OT hours, bonuses, or manually entered amounts for the employee are silently ignored. Operator may not notice until they compare payslips to the timesheet.

Mitigation: The readiness check surfaces a count. The operator should delete the orphaned inputs manually before closing the period. No automatic cleanup exists.

Surfaced at: backend/api/routes/payroll_input.py:270–317
3
contract_start before salary_definition.effective_from
Architect Security Guarded
▼
Setup
salary_definition.effective_from = '2026-02-01' payload.contract_start = '2026-01-15' # before effective_from
Current Behaviour

422 error returned: "Salary definition 'X' is not effective until 2026-02-01 — cannot use for a contract starting 2026-01-15"

Guard exists at the API boundary in both workspace.py::create_employee and employees.py::add_contract.

Risk & Mitigation

Remaining risk: The guard is only at the API layer. If salary_definitions are created retroactively in the DB (direct INSERT), the payroll engine will encounter a contract whose period precedes the salary definition — the engine's behaviour is undefined in this case (it uses whatever is in the frozen snapshot, which may be empty).

Mitigation: API guard covers all legitimate paths. DB-level CHECK constraint (effective_from on salary_definition as a partial index) would close the direct-DB path — not yet implemented.

Guard at: backend/api/routes/workspace.py:304–314, employees.py:54–74
4
ITF threshold races the year boundary
Auditor Architect High
▼
Setup
# Running January payroll on Dec 31 period_end = '2027-01-31' # new year # ITF YTD query uses EXTRACT(YEAR FROM period_end) # Returns 2027 — YTD sum = ₦0 (no 2027 APPROVED runs yet) # ITF threshold not met → employer skips ITF this run
Current Behaviour

ITF YTD query filters EXTRACT(YEAR FROM period_end) = EXTRACT(YEAR FROM :period_end). On the first run of a new calendar year, YTD = ₦0 and employee count from the prior year is not included.

ITF threshold check may fail even if the organisation was well above ₦50M annual payroll in the prior year.

Risk & Mitigation

Risk: ITF employer cost (1% of BASIC + HOUSING + TRANSPORT) is incorrectly skipped in the first payroll run of a new calendar year, understating employer cost obligations. Auditors checking ITF contributions will see a January gap.

Mitigation: Currently unmitigated. Fix: use a rolling 12-month window rather than calendar year, or seed a YTD carry-forward record at year-start. Logged as a future improvement item.

Query at: backend/api/routes/payroll.py:739–753
5
Two workspace rules injecting the same component name
Architect Auditor High
▼
Setup
# Two active payroll rules, same rule_name rule_1 = {"rule_name": "OVERTIME_HOURS", ...} rule_2 = {"rule_name": "OVERTIME_HOURS", ...} # Both apply_payroll_rules() and sequential_executor # use dict — last write wins, no error, no warning
Current Behaviour

The second rule silently overwrites the first in salary_components dict. No exception is thrown. The trace will show both rules as "applied" but only the second amount persists.

The first rule's amount is lost with no indication in the output.

Risk & Mitigation

Risk: OT or bonus amounts silently halved or replaced. An operator configuring rules via the UI could accidentally create two rules with the same output code. There is no uniqueness constraint on rule_name in the payroll_rule table.

Mitigation: Currently no DB-level guard. Mitigation: add UNIQUE constraint on (workspace_id, rule_name) in payroll_rule table (future sprint). The UI should also warn if a duplicate rule_name is detected during rule creation.

Behaviour in: backend/domain/payroll/rule_evaluator.py (dict overwrite pattern)
6
Legacy scalar input bypasses temporal rate resolution
Architect Low (legacy path)
▼
Setup
# Old callers pass a bare number, not a list of events employee_inputs = {"OT_HOURS": 8.0} # scalar # _to_events() normalises to: # [{quantity: 8.0, reference_date: None}] # reference_date=None → uses CURRENT rule, not historical
Current Behaviour

The scalar is normalised into a single event with reference_date=None. The temporal resolution logic skips historical lookup for null reference dates and uses the current rule set.

If the OT rate changed mid-period, this input gets the current rate, not the rate that was effective when the OT occurred.

Risk & Mitigation

Risk: Only affects retroactive or mid-period inputs passed via old code paths (bulk upload legacy format, simulation scripts). The production API path always sends events with reference_date. Low risk for live runs; high risk for back-pay calculations using the simulate script.

Mitigation: Migrate all callers to the event format [{quantity, reference_date}]. Simulation script still uses the legacy format — see Track Q item Q3.

Normalisation at: backend/domain/payroll/rule_evaluator.py::_to_events()
7
Daily rate deduction on a zero-working-day period
Architect Handled (silent skip)
▼
Setup
# Pay period with zero working days # (suspension, all-PH month, misconfiguration) working_days = 0 proration_strategy = "work_days" # divisor = 0 → would cause ZeroDivisionError
Current Behaviour

The rule_evaluator's daily_rate_deduction handler detects the zero divisor and silently skips the deduction for affected components. No exception is raised. A trace entry is written with status="not_applied" and note="zero divisor for strategy work_days".

Risk & Mitigation

Risk: If a period genuinely has 0 working days (e.g. a national holiday falls on every working day), absence deductions are skipped and employees receive full pay. This is arguably correct for a zero-working-day period, but if it's a data error (wrong working_days_override), the operator gets no visible warning that deductions were skipped.

Mitigation: The trace entry is visible in the component trace. A pre-flight check warning for zero working_days would surface this earlier. Not yet implemented.

Guard at: backend/domain/payroll/rule_evaluator.py (daily_rate_deduction handler)
8
PAYE_ONLY input adds to taxable income but not gross or net
Operator Auditor By design — confusing
▼
Setup
# LTA, non-cash benefit, notional benefit in kind payroll_input.input_category = 'PAYE_ONLY' # Flows into TAXABLE_INCOME only # Does NOT increase GROSS_PAY # Does NOT increase NET_PAY # DOES increase PAYE deduction
Current Behaviour

The amount is added to TAXABLE_INCOME at priority 95 (PAYE_ONLY_ADDITIONS component). GROSS_PAY and NET_PAY are not affected. PAYE is computed on the higher taxable income, so the employee pays more PAYE but receives the same net pay.

Risk & Mitigation

Risk: Operators and auditors comparing the payslip may notice that gross + deductions ≠ net in the expected way. The PAYE_ONLY amount increases PAYE but the "income" that caused it is not visible in gross pay. Payslips need a dedicated line for PAYE_ONLY additions to make this transparent.

This is correct legal treatment for non-cash benefits in Nigeria (e.g. LTA, housing allowance classified as benefit-in-kind). The confusion is a communication problem, not a calculation bug.

Mitigation: Payslip template (when built) must include a "Benefits in Kind" section showing PAYE_ONLY amounts separately from gross earnings.

Handled at: sequential_executor.py priority 95 (_handle_paye_only_additions)
9
PARTIAL run retried after salary definition change
Architect Auditor By design — safe
▼
Setup
# Run created (snapshot frozen) # 3 employees FAILED → PARTIAL # Operator edits salary definition components # Operator retries FAILED employees
Current Behaviour

The retry uses the frozen snapshot from the original DRAFT creation, not the live salary definition. The edited salary definition has no effect on this run. The 3 failed employees are retried with the original component amounts.

The edit-lock (D-ARCH-1) also blocks salary definition changes while the run is in APPROVED or later states.

Risk & Mitigation

This is the correct and safe behaviour. The snapshot freeze guarantees that all employees in the same run use the same component definitions, regardless of how many retries are needed. Without this invariant, employees processed in the first pass and employees retried later could use different salary amounts within the same run.

Operator communication: If the operator genuinely needs the salary definition change to take effect, they must cancel the current run and create a new DRAFT. The UI should make this clear.
